# 日記帳分錄測試 (Journal Entry Testing) — 領域知識模型

## 文件說明

本文件為 Journal Entry Testing (JET) 的完整領域知識模型，記錄了日記帳分錄測試的業務邏輯、資料模型、審計流程與篩選規則。本文件為平台無關 (platform-agnostic) 的業務規格書，不綁定任何特定實作技術。

本文件的知識來源為台灣本地審計服務中，基於 Caseware IDEA 所開發的 JET 工具實務經驗，並整合後續 MVP 重構過程中的領域分析成果。

---

## 1. 業務背景與目的

### 1.1 什麼是日記帳分錄測試 (JET)

日記帳分錄測試 (Journal Entry Testing, JET) 是一項**實質性審計程序 (Substantive Audit Procedure)**，依據國際審計準則 ISA 240《財務報表審計中對舞弊之考量》與 ISA 330《審計人員對所評估風險之因應》的要求，針對**管理階層凌駕控制 (Management Override of Controls)** 的風險，對總帳中的日記帳分錄 (Journal Entries) 及其他調整進行測試。

### 1.2 為什麼需要 JET

管理階層因其職位與職權，具備凌駕內部控制的能力，可能透過以下方式操縱財務報表：

- 記錄虛構或不當的日記帳分錄（特別是在期末結帳前後）
- 對財務報表金額進行不當的估計調整
- 隱匿或延遲認列交易

JET 的目的是透過**全母體分析 (Full Population Analysis)** 的方式，從企業整個會計期間的所有日記帳分錄中，依據風險導向的篩選條件，辨識出高風險或異常的分錄，供審計人員進一步調查。

### 1.3 JET 的核心邏輯

```
總帳 (GL) + 試算表 (TB)
    ↓ 匯入與驗證
資料驗證（完整性、借貸平衡、空值檢查）
    ↓ 設定輔助檔案
科目配對表 + 假日曆 + 補班日曆
    ↓ 預篩選程序
8 項標準篩選 (R1-R8) + 自訂擴充 (A2-A4)
    ↓ 進階篩選
科目配對分析 + 自訂條件
    ↓ 產出
審計工作底稿
```

## 2. 資料實體模型

### 2.1 總帳分錄 (General Ledger / Journal Entry)

總帳分錄是 JET 的**主要分析對象**，每一筆記錄代表一張傳票中的一個分錄行 (line item)。

#### 核心欄位

| 欄位名稱 | 英文邏輯名 | 說明 | 必填 |
|:---|:---|:---|:---|
| 傳票號碼 | DocumentNumber | 傳票的唯一識別碼，同一張傳票下可有多個分錄行 | 是 |
| 傳票文件項次 | LineItem | 同一傳票內的分錄序號 | 否 |
| 傳票金額 | Amount | 分錄金額（正數為借方，負數為貸方） | 是 |
| 會計科目編號 | AccountCode | 總帳科目代號 | 是 |
| 會計科目名稱 | AccountName | 總帳科目名稱 | 否 |
| 傳票摘要 | DocumentDescription | 分錄說明文字 | 否 |
| 傳票核准日 | ApprovalDate | 傳票核准的日期 | 否 |
| 總帳日期 | PostDate | 過帳至總帳的日期 | 否 |
| 傳票建立人員 | CreatedBy | 編製此傳票的人員 | 否 |
| 傳票核准人員 | ApprovedBy | 核准此傳票的人員 | 否 |
| 分錄來源模組 | SourceModule | 產生此分錄的子系統 (如 AP, AR, GL) | 否 |
| 人工傳票否 | IsManual | 標記是否為人工輸入的傳票 (0=自動, 1=人工) | 否 |

#### 衍生欄位（處理過程中計算產生）

| 衍生欄位 | 計算規則 | 用途 |
|:---|:---|:---|
| 借方金額 (DebitAmount) | `IF Amount >= 0 THEN Amount ELSE 0` | R5, R6 彙總分析 |
| 貸方金額 (CreditAmount) | `IF Amount < 0 THEN Amount ELSE 0` | R5, R6 彙總分析 |
| 借貸方向 (DrCr) | `IF Amount >= 0 THEN "DEBIT" ELSE "CREDIT"` | 分類標記 |

#### 金額處理方式

不同企業的 ERP 系統對金額的記錄方式不同，JET 需要支援以下幾種模式：

| 模式 | 說明 |
|:---|:---|
| 模式 0 (單一金額欄) | 正數 = 借方、負數 = 貸方，最常見 |
| 模式 1 (借貸方向欄) | 金額欄為絕對值，另有「借貸別」欄位標記方向 |
| 模式 2 (借方標誌欄) | 金額欄為絕對值，另有「是否借方」欄位 (0/1) |
| 模式 3 (雙金額欄) | 分別有「借方金額」和「貸方金額」兩個欄位 |

### 2.2 試算表 (Trial Balance)

試算表是各會計科目在特定期間的餘額彙總，用於與總帳分錄進行**完整性比對 (Completeness Test)**。

#### 核心欄位

| 欄位名稱 | 英文邏輯名 | 說明 | 必填 |
|:---|:---|:---|:---|
| 會計科目編號 | AccountCode | 與 GL 對應的科目代號 | 是 |
| 會計科目名稱 | AccountName | 科目名稱 | 否 |
| 期間變動金額 | ChangeAmount | 本期間的淨變動額 | 視模式 |
| 期初金額 | OpeningBalance | 期初餘額 | 視模式 |
| 期末金額 | ClosingBalance | 期末餘額 | 視模式 |
| 期初借方金額 | OpeningDebitBalance | 期初借方餘額 | 視模式 |
| 期初貸方金額 | OpeningCreditBalance | 期初貸方餘額 | 視模式 |
| 期末借方金額 | ClosingDebitBalance | 期末借方餘額 | 視模式 |
| 期末貸方金額 | ClosingCreditBalance | 期末貸方餘額 | 視模式 |
| 借方金額 | DebitAmount | 本期借方合計 | 視模式 |
| 貸方金額 | CreditAmount | 本期貸方合計 | 視模式 |

#### TB 金額計算模式

試算表依照來源系統提供的欄位不同，有四種計算 TB 變動金額的方式：

| 模式 | 可用欄位 | 計算方式 |
|:---|:---|:---|
| 模式 1 | 僅變動金額 | 直接使用 ChangeAmount |
| 模式 2 | 期初 + 期末 | ChangeAmount = ClosingBalance - OpeningBalance |
| 模式 3 | 借方 + 貸方 | ChangeAmount = DebitAmount - CreditAmount |
| 模式 4 | 期初借貸 + 期末借貸 | ChangeAmount = (ClosingDebit - ClosingCredit) - (OpeningDebit - OpeningCredit) |

### 2.3 科目配對表 (Account Mapping)

科目配對表將企業的會計科目對應至**標準化科目分類**，用於預篩選程序 R3（未預期借貸組合）的判斷。

| 欄位名稱 | 英文邏輯名 | 說明 |
|:---|:---|:---|
| 科目編號 | AccountCode | 對應 GL 的會計科目編號 |
| 科目名稱 | AccountName | 科目名稱 |
| 標準化分類 | AccountStandardized | 標準化科目分類名稱 |

#### 標準化科目分類

| 分類名稱 | 用途 |
|:---|:---|
| Revenue | 收入科目 — 用於 R3 識別異常貸方 |
| Receivables | 應收帳款科目 — 用於 R3 識別異常借方 |
| Cash | 現金科目 — 用於 R3 識別異常借方 |
| Receipt in advance | 預收款項科目 — 用於 R3 識別異常借方 |
| Others | 其他未分類科目 |

> **說明**: 以上為預設分類。科目配對表為可設定的 (configurable)，審計人員可依據案件特性自訂分類。

### 2.4 日期維度表 (Date Dimension)

日期維度表為一張日曆主檔，涵蓋審計期間的每一天，標註假日與補班日資訊，用於 R7（週末）和 R8（假日）篩選。

| 欄位名稱 | 英文邏輯名 | 說明 |
|:---|:---|:---|
| 日期主鍵 | DateKey | 格式 YYYYMMDD 的唯一識別值 |
| 完整日期 | FullDate | 日期欄位 |
| 年 | Year | 西元年 |
| 月 | Month | 月份 (1-12) |
| 日 | Day | 日 (1-31) |
| 星期幾 | DayOfWeek | 1=日, 2=一, 3=二, 4=三, 5=四, 6=五, 7=六 |
| 是否週末 | IsWeekend | 系統自動依星期幾判定 |
| 是否假日 | IsHolidays | 由使用者上傳的假日曆覆寫 |
| 是否補班日 | IsMakeupDays | 由使用者上傳的補班日曆覆寫 |
| 假期說明 | HolidaysDesc | 假日名稱（選填） |
| 補班日說明 | MakeupDaysDesc | 補班日說明（選填） |

## 3. 審計工作流程

JET 的完整審計流程分為五個步驟，每個步驟有明確的輸入、處理邏輯與產出。

### 3.1 步驟一：資料匯入與驗證 (Data Import & Validation)

**目的**: 將受查企業的總帳 (GL) 和試算表 (TB) 資料匯入系統，完成欄位映射，並執行基礎資料驗證。

**輸入**:
- 企業匯出的 GL 檔案 (CSV / Excel)
- 企業匯出的 TB 檔案 (CSV / Excel)

**處理邏輯**:
1. **欄位映射** — 使用者將原始檔案的欄位對應至標準化欄位名稱
2. **資料型別驗證** — 確認金額欄位為數值型、日期欄位格式正確
3. **空值檢查** — 檢測關鍵欄位 (科目編號、傳票號碼、摘要) 的空值紀錄
4. **完整性測試 (Completeness Test)** — 核對 GL 分錄加總與 TB 變動額是否一致
5. **借貸平衡測試 (Document Balance Test)** — 驗證每張傳票的借貸合計為零
6. **INF 抽樣測試** — 隨機抽取樣本供人工檢查

**產出**:
- 已匯入且標準化的 GL/TB 資料
- 驗證報告 (Validation Report)
- INF 測試報告 (INF Report)

### 3.2 步驟二：輔助檔案設定 (Configuration)

**目的**: 上傳並設定篩選程序所需的輔助資料。

**輸入**:
- **科目配對表** — 將 GL 科目編號對應至標準化分類 (Revenue, Receivables, Cash 等)
- **假日曆** — 該會計年度的國定假日清單
- **補班日曆** — 因假日調整而需補班的工作日清單

**處理邏輯**:
1. 驗證科目配對表格式，建立科目 → 分類的查詢表
2. 驗證假日曆日期格式，建立日期 → 假日名稱的查詢表
3. 驗證補班日曆，標記假日中實為工作日的日期
4. 建立日期維度表，合併週末、假日、補班日資訊

**產出**:
- 科目配對對照表
- 日期維度表（含週末、假日、補班日標記）

### 3.3 步驟三：預篩選程序 (Pre-Screening Routines)

**目的**: 依據 8 項標準化風險指標與最多 3 項自訂指標，對全母體 GL 資料進行風險篩選，標記符合條件的分錄。

詳見 [第 5 節：預篩選程序詳述](#5-預篩選程序詳述-r1-r8--a2-a4)。

**輸入**:
- 已驗證的 GL 資料
- 科目配對表、日期維度表
- 使用者設定的自訂參數 (自訂關鍵字、自訂尾數位數等)

**處理邏輯**:
1. 前置處理 — 計算借方/貸方金額衍生欄位
2. 建立傳票層級彙總表 (按傳票號碼彙總)
3. 依序執行 R1-R8 及 A2-A4 篩選程序
4. 每項篩選結果以標記欄位 (PRESCR_Rx = "Y") 附加至彙總表

**產出**:
- 各篩選程序的結果檔案 (R1-R8, A2-A4)
- 附帶所有篩選標記的傳票彙總表

### 3.4 步驟四：進階篩選 (Advanced Filtering)

**目的**: 審計人員從預篩選結果中，依據專業判斷組合多項條件，決定最終測試母體。

詳見 [第 6 節：進階篩選邏輯](#6-進階篩選邏輯)。

**輸入**:
- 預篩選結果 (含 R1-R8, A2-A4 標記)
- 審計人員選擇的篩選條件組合
- 自訂的科目配對條件

**處理邏輯**:
1. 根據審計人員選擇的篩選條件組合過濾母體
2. 執行科目配對分析（三種配對模式）
3. 支援額外的日期區間、金額區間、文字比對等自訂條件

**產出**:
- 篩選後的測試母體 (Filtered JE Population)

### 3.5 步驟五：工作底稿產出 (Work Paper Generation)

**目的**: 將篩選結果整理為正式的審計工作底稿，供存檔與覆核。

**輸入**:
- 篩選後的測試母體
- 專案元資料 (客戶、期間、編製人等)

**處理邏輯**:
1. 產生案件總覽 (Engagement Overview)
2. 產生資料總覽 (Data Overview) — GL/TB 的筆數與金額統計
3. 產生驗證總覽 (Validation Overview) — 各項驗證異常的計數
4. 輸出完整性差異明細 (Completeness Detail)
5. 輸出借貸不平差異明細 (Document Balance Detail)
6. 輸出 INF 抽樣明細 (INF Sample Detail)
7. 輸出各預篩選程序的明細與彙總底稿
8. 輸出科目配對資訊與欄位映射資訊

**產出**:
- 審計工作底稿 (Excel 報表)，包含以下工作表：
  - Engagement Overview
  - Data Overview
  - Validation Overview
  - Completeness Detail
  - Document Balance Detail
  - INF Sample Detail
  - R1-R8 / A2-A4 各程序底稿
  - Account Mapping Info
  - Field Mapping Info

## 4. 資料驗證規則

### 4.1 完整性測試 (Completeness Test / Roll-forward Test)

**目的**: 確認匯入的 GL 分錄母體是否完整，即 GL 按科目加總後的金額是否與 TB 的期間變動額一致。

**審計意義**: 如果 GL 與 TB 有差異，代表匯入的分錄母體可能不完整（有遺漏），測試結果將不具代表性。

**規則邏輯**:

```
1. 按科目加總 GL 金額:
   JE_ACCOUNT_SUM = GROUP BY AccountCode → SUM(Amount)

2. 與 TB 做 FULL OUTER JOIN (以 AccountCode 為鍵):
   TB ←全連結→ JE_ACCOUNT_SUM

3. 計算差異:
   DIFF = TB.ChangeAmount - JE_ACCOUNT_SUM.AmountSum

4. 合併科目編號 (處理單邊存在的情況):
   ACCOUNT_MERGED = IF JE.AccountCode 非空 THEN JE.AccountCode ELSE TB.AccountCode

5. 篩選異常:
   WHERE ABS(DIFF) > 0
```

**輸出**: 差異科目清單，包含科目編號、科目名稱、TB 變動金額、GL 加總金額、差異金額。

### 4.2 借貸平衡測試 (Document Balance Test)

**目的**: 驗證每一張傳票的借方金額合計與貸方金額合計是否平衡（淨額為零）。

**審計意義**: 會計複式簿記的基本原理要求每張傳票借貸必須平衡。不平衡的傳票可能代表資料品質問題或異常。

**規則邏輯**:

```
1. 按傳票號碼加總:
   DOC_SUM = GROUP BY DocumentNumber → SUM(Amount)

2. 篩選不平衡傳票:
   WHERE DOC_SUM.AmountSum ≠ 0

3. 取得不平衡傳票的所有分錄明細:
   GL INNER JOIN DOC_SUM_DIFF ON DocumentNumber

4. 產生明細報表:
   - 按傳票號碼與總帳日期彙總
   - 分別計算借方合計與貸方合計
```

**輸出**: 不平衡傳票清單，包含傳票號碼、總帳日期、借方合計、貸方合計。

### 4.3 INF 抽樣測試 (INF Sampling Test)

**目的**: 從 GL 母體中隨機抽取樣本，供審計人員執行人工檢查，驗證非財務性資訊欄位 (如科目名稱、摘要、日期) 的可靠性。

**審計意義**: 在依賴 GL 資料進行風險篩選之前，需先確認用於篩選的非財務性欄位是否準確可靠（即 INF — Information produced by the entity）。

**規則邏輯**:

```
1. 以審計期間的 GL 為母體
2. 隨機抽取 60 筆記錄
3. 按傳票號碼排序
```

**輸出**: 60 筆隨機樣本，供人工核對各欄位正確性。

> **說明**: 抽樣數量 60 為工具預設值，可依審計判斷調整。

### 4.4 空值紀錄測試 (Null Records Test)

**目的**: 檢查 GL 資料中關鍵欄位的空值紀錄。

**審計意義**: 空值紀錄可能導致後續篩選程序遺漏高風險分錄。

**規則邏輯**:

```
1. 篩選科目編號為空: WHERE AccountCode IS NULL
2. 篩選傳票號碼為空: WHERE DocumentNumber IS NULL
3. 篩選傳票摘要為空: WHERE DocumentDescription IS NULL
```

**輸出**: 三類空值紀錄的個別清單與計數。

## 5. 預篩選程序詳述 (R1-R8 + A2-A4)

每項預篩選程序對應一個已知的舞弊風險指標。執行後，符合條件的分錄會被標記 (Tag)，並可在步驟四中組合使用。

### 5.1 R1: 期末財報準備日後核准之分錄

| 項目 | 內容 |
|:---|:---|
| **風險指標** | 在財報準備期間之後才核准的分錄，可能是為了操縱期末財務數字 |
| **篩選邏輯** | `ApprovalDate >= LastAccountingPeriodDate` |
| **依賴條件** | 需有「傳票核准日」欄位；需設定「期末最後會計日」 |
| **標記欄位** | PRESCR_R1 = "Y" |

### 5.2 R2: 分錄摘要出現特定描述

| 項目 | 內容 |
|:---|:---|
| **風險指標** | 摘要中包含暗示調整、沖銷、錯誤等異常性質的關鍵字 |
| **篩選邏輯** | 以正則表達式 (Regex) 比對大寫且去除空白後的摘要欄位 |
| **預設關鍵字** | 見 [附錄 B](#附錄-b-r2-預設關鍵字清單) |
| **依賴條件** | 需有「傳票摘要」欄位 |
| **標記欄位** | PRESCR_R2 = "Y" |

### 5.3 R3: 未預期出現之特定借貸組合

| 項目 | 內容 |
|:---|:---|
| **風險指標** | 同一張傳票中出現不合常理的借貸科目組合，可能暗示虛假收入或挪用資金 |
| **篩選邏輯** | 步驟 1: 將 GL 與科目配對表連結，取得每筆分錄的標準化分類 |
|  | 步驟 2: 篩選出「Revenue 科目 + 貸方 (Amount < 0)」的分錄 |
|  | 步驟 3: 篩選出「Receivables / Cash / Receipt in advance 科目 + 借方 (Amount > 0)」的分錄 |
|  | 步驟 4: 以傳票號碼配對，找出同時存在步驟 2 與步驟 3 的傳票 |
| **依賴條件** | 需上傳科目配對表，且必須至少包含 Revenue 類與 Receivables/Cash/Receipt in advance 類 |
| **標記欄位** | PRESCR_R3 = "Y" |
| **未設定科目配對表時** | 此程序不執行，標記為 N/A |

> **業務邏輯說明**: R3 的核心思路是偵測「認列收入 (Credit to Revenue) 同時增加資產 (Debit to Receivables/Cash)」的傳票。這是虛假收入最常見的分錄模式。

### 5.4 R4: 分錄金額中有連續零的尾數 (整數金額)

| 項目 | 內容 |
|:---|:---|
| **風險指標** | 整數金額（尾數為大量零）的分錄可能代表估計值或人為捏造的金額 |
| **篩選邏輯** | 步驟 1: 計算全母體借方金額的平均值位數 N |
|  | 步驟 2: 篩選金額整數部分末尾 N-1 位全為零的分錄 |
|  | 例: 若平均借方金額為 5 位數，則篩選尾數 4 個零的分錄 (如 10,000, 50,000) |
| **標記欄位** | PRESCR_R4 = "Y" |

> **說明**: 尾數零的位數為動態計算，與母體金額規模成正比。使用者亦可透過 A4 自訂尾數位數。

### 5.5 R5: 依分錄編製者彙總分錄

| 項目 | 內容 |
|:---|:---|
| **風險指標** | 分錄集中在少數人員編製，可能代表職能分離 (Segregation of Duties) 不足 |
| **篩選邏輯** | 按「傳票建立人員」分群，彙總各人員的借方合計、貸方合計、筆數 |
|  | 若有「人工傳票否」欄位，則進一步細分人工/自動分錄 |
| **依賴條件** | 需有「傳票建立人員」欄位 |
| **產出** | 彙總報表（非二元標記），供審計人員判讀 |

> **說明**: R5 與其他篩選程序不同，它產出的是**彙總分析報表**而非標記個別分錄。審計人員需根據彙總結果，自行判斷是否需要針對特定人員進一步調查。

### 5.6 R6: 較少使用之科目

| 項目 | 內容 |
|:---|:---|
| **風險指標** | 使用頻率極低的科目可能被用來隱藏不當分錄 |
| **篩選邏輯** | 按「會計科目編號 + 會計科目名稱」分群 |
|  | 彙總各科目的筆數、借方合計、貸方合計 |
|  | 按筆數遞增排序 (使用次數最少的排在最前面) |
| **產出** | 彙總報表，供審計人員判讀 |

> **說明**: 與 R5 相同，R6 產出的是彙總報表。何謂「低頻率」由審計人員依專業判斷決定。

### 5.7 R7: 週末過帳/核准之分錄

| 項目 | 內容 |
|:---|:---|
| **風險指標** | 在非工作日 (週末) 過帳或核准的分錄為異常，正常企業運作不應在週末處理傳票 |
| **篩選邏輯** | 步驟 1: 計算傳票核准日/總帳日期的星期幾 |

| **標記欄位** | DOC_WEEKEND_JE_T, POST_WEEKEND_JE_T |

### 5.8 R8: 假日過帳/核准之分錄

## 6. 進階篩選邏輯

步驟四讓審計人員在預篩選結果的基礎上，進一步精煉測試母體。

### 6.1 科目配對分析 (Account Pairing Analysis)

科目配對分析用於找出特定借方科目與特定貸方科目「同時出現在同一張傳票」的分錄。提供三種配對模式：

#### Type A: 精確配對

```
找出同時滿足以下條件的傳票：
- 包含指定借方科目分類 (Amount >= 0) 的分錄
- 包含指定貸方科目分類 (Amount < 0) 的分錄
輸出：該傳票中所有符合上述兩類條件的分錄行
```

**使用時機**: 已明確知道可疑的借方與貸方科目組合時。

#### Type B: 借方錨定

```
步驟 1: 找出包含指定借方科目分類 (Amount >= 0) 的傳票
步驟 2: 取出這些傳票中所有的貸方分錄 (Amount < 0)
輸出：借方分錄 + 同張傳票的所有貸方分錄
```

**使用時機**: 知道可疑的借方科目，想了解「對方科目」是什麼。

#### Type C: 貸方錨定

```
步驟 1: 找出包含指定貸方科目分類 (Amount < 0) 的傳票
步驟 2: 取出這些傳票中所有的借方分錄 (Amount >= 0)
輸出：貸方分錄 + 同張傳票的所有借方分錄
```

**使用時機**: 知道可疑的貸方科目，想了解「對方科目」是什麼。

### 6.2 自訂條件篩選

除了科目配對外，審計人員還可疊加以下篩選條件：

| 條件類型 | 說明 |
|:---|:---|
| 預篩選標記組合 | 選取一或多項 R1-R8 / A2-A4 的標記 |
| 日期區間 | 限定核准日期或總帳日期的範圍 (最多 2 組) |
| 金額區間 | 限定分錄金額的範圍 |
| 文字比對 | 依特定欄位值進行文字匹配 (最多 2 組) |
| 借方/貸方限定 | 僅選取借方或貸方分錄 |
| 人工/自動分錄 | 僅選取人工傳票或自動傳票 |
| 期內/期外 | 限定於會計期間內或期間外的分錄 |

## 8. 台灣在地化考量

### 8.1 假日與補班日制度

台灣特有的「彈性假日」與「補班日」制度會影響 R7 和 R8 的判斷：

- **彈性假日**: 當國定假日遇到週二或週四，前後的週一或週五會調整為放假日
- **補班日**: 因彈性假日而額外指定的週六上班日

因此：
- R7（週末篩選）需排除補班日的週六 — 該日實為工作日
- R8（假日篩選）需包含因彈性假日而放假的工作日

### 8.2 中文關鍵字

R2 的預設關鍵字清單包含台灣審計實務中常見的中文異常描述用語，詳見 [附錄 B](#附錄-b-r2-預設關鍵字清單)。

### 8.3 會計科目命名

台灣企業的會計科目多以中文命名，科目編號則依循財政部或各行業公會的編碼規範。科目配對表需對應至英文標準化分類。
