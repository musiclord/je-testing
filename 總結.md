## 1. 問題背景與目標

- 既有工具原本建立在 **Caseware IDEA + VBA** 上。
- 因為 **不再訂閱 IDEA**，所以原本的 JET（Journal Entry Testing）工具需要尋找替代方案。
- 現有可評估方案中，**MindBridge AI AITS** 雖然可做風險管理與異常偵測，但**不足以滿足本地審計實務需要的細緻篩選條件定義**，例如：
  - 自定 JE 借貸規則矩陣
  - 特定欄位條件組合
  - 本地化的業務邏輯與規則例外
- 因此核心需求不是單純「買一套 AI 審計工具」，而是建立一個**可中心化管理、能處理大量 GL/TB、可由審計員靈活設定篩選條件的 JET 系統**。

## 2. 核心業務需求

### 2.1 主要資料處理需求
- 上傳各客戶不同格式的：
  - **General Ledger (GL)**
  - **Trial Balance (TB)**
- 支援資料量從：
  - 數千筆
  - 到數千萬筆
- 必須做：
  - 欄位 mapping
  - 資料標準化
  - 規則篩選
  - JE 測試結果輸出

### 2.2 主要功能目標
- 審計員透過簡單介面：
  - 上傳 GL / TB
  - 選擇或輸入篩選條件
  - 執行 JET 規則
  - 查看結果
  - 匯出 Excel / working paper
- 核心在於：
  - **把審計員的篩選條件轉成後端可執行邏輯**
  - 並讓 **SQL Server** 做真正的大量運算

### 2.3 實務限制
- 不走內網 web server 形式
- 盡量避免額外安裝與資安通報風險
- 最理想是：
  - **打包成單一 `.exe`**
  - 使用者直接執行
  - 不必自行安裝複雜依賴
- 公司環境：
  - **Python 不可作為正式方案**（全球資安規範）
- 使用者與審計團隊偏好：
  - Windows / Office 生態
  - 可與 SQL Server 整合
  - UI 只需服務核心功能，不需過度花俏

## 4. 前端與後端分工的最終理解

## 4.1 核心原則
前端只負責：
- 操作介面
- 條件輸入
- 顯示結果
- 上傳檔案
- 觸發動作

後端負責：
- 驗證參數
- 執行規則
- 轉換查詢
- 呼叫 SQL Server
- 整理結果

資料庫負責：
- ETL
- 標準化
- stored procedures / views
- rule tables
- 結果表

## 4.2 重要架構原則
**前端不要直接拼 raw SQL。**

正確方式應是：
1. 前端送出 action + payload
2. `.NET` service layer 驗證並轉成參數化查詢或 stored procedure 呼叫
3. SQL Server 執行運算
4. `.NET` 回傳結果給前端

## 5. WinForms、WPF、Blazor Hybrid、WebView2 的比較結論

## 5.1 WPF
### 優點
- 資料繫結較完整
- UI / 邏輯分離較好
- styles / templates / XAML 能力較強
- 中長期擴充性較高

### 缺點
- Windows-only
- 在 macOS 上不能原生接續開發 UI
- 對目前 Phase 1 的 JET 核心需求而言，可能過重
- 若正式前端最終由 HTML 接手，WPF 原本的 UI 優勢會被削弱

### 結論
- 適合較複雜、長期擴充的大型桌面系統
- 但對目前 JET Phase 1 並非最划算起點

---

## 5.2 WinForms
### 優點
- 最符合現在的 Phase 1 需求
- 適合 data-driven / line-of-business desktop apps
- 表單、按鈕、DataGridView、匯出這類操作很直觀
- 容易快速落地
- 比 WPF 更輕量
- 較適合在不追求複雜桌面 UI 的情況下快速實現

### 缺點
- UI 上限與客製化能力低於 WPF
- 純原生 WinForms UI 不如 HTML 好看、好做

### 結論
若只聚焦於：
- 上傳 GL / TB
- 勾選條件
- 送出查詢
- 顯示結果
- 匯出 Excel

那麼 **WinForms 是最務實的桌面 host**。

---

## 5.3 Blazor Hybrid
### 優點
- 前端寫法接近 HTML / web component
- AI 容易協助產 UI
- 適合喜歡 web-style UI 的開發模式

### 缺點
- 本質上仍依賴 WebView
- 對正式產品來說多一層 runtime / host 考量
- 在你這個「單 exe、少安裝、少資安摩擦」情境下，不如 WinForms/WPF 純桌面 host 穩

### 結論
- 適合作為概念方案
- 但不再是正式產品首選

---

## 5.4 WebView2
### 角色定位
WebView2 不是整個系統，而是：
- 把 **HTML/CSS/JS** 嵌進 WinForms / WPF 桌面程式的橋接層

### 優點
- 讓 AI 生成的 HTML 前端可以直接變成正式 UI
- 支援拖曳上傳、日期選擇器、互動表格、動態篩選器
- 保留桌面應用形式，不需架 web server
- 你已確認 **Windows 本機已有 WebView2 Runtime**

### 缺點
- 若目標機器沒有 runtime 會增加部署考量
- 仍需管理前端與 .NET 的橋接方式

### 結論
在已確認本機有 WebView2 Runtime 的前提下，  
**WinForms + WebView2** 的可行性大幅提高。

## 6. HTML 前端是否可取代原生 UI？

## 6.1 已確認的結論
**可以，且對你的情境非常合理。**

但精準說法是：

- HTML 取代的是 **前端畫面層**
- 不是取代整個應用主體
- 也不是直接讓 HTML 連 SQL Server

### 最佳結構
**WinForms host**
→ **WebView2**
→ **本地 HTML/CSS/JS 前端**
→ **.NET methods / bridge**
→ **SQL Server**

### HTML 前端可負責
- 拖曳 / 點擊上傳 GL、TB
- 日期選擇
- 動態條件輸入
- 表格結果預覽
- 工作流程步驟 UI
- 篩選器面板

### `.NET` 負責
- 將前端動作綁定到具體方法
- 呼叫 SQL Server
- 管理參數驗證
- 管理匯入與匯出流程

### 結論
如果你的重點是：
- 讓 AI 盡量接手前端
- 用 HTML 快速驗證與實作 UI
- 同時保持本地 exe + SQL Server 架構

那麼 **HTML 前端 + WinForms host + WebView2 + .NET backend** 是高可行性方案。

## 7. Mac 與 Windows 雙機開發的結論

## 7.1 WPF 在 macOS 的狀況
- WPF 是 Windows-only
- 不可在 macOS 原生完整開發
- Visual Studio for Mac 已退役
- 社群若不得已要在 Mac 碰 WPF，常見做法是：
  - Windows VM（例如 Parallels）
  - 遠端 Windows 開發機
  - Mac 只負責非 UI 部分

## 7.2 若最終採 WinForms + WebView2
這就降低了 WPF 的問題：
- WinForms host 主要在 Windows 端整合即可
- HTML 前端與大部分 `.NET` 邏輯仍可在跨平台工具下編寫
- Mac 可用來做：
  - HTML POC
  - SQL scripts
  - service layer
  - DTO / models
  - tests
  - AI 驅動的 repo 級開發
- Windows 則負責：
  - WinForms host
  - WebView2 整合
  - 最終打包與正式驗證

## 10. C#、VB、.NET Framework、.NET 10 的結論

## 10.1 語言
**選 C#，不要選 VB。**

原因：
- 社群資源較多
- AI 生成與範例品質更好
- 未來維護與招人都較容易
- 現代 .NET 生態以 C# 為主

## 10.2 執行時與框架
**選現代 .NET（.NET 10 LTS）**
不要選：
- 舊 .NET Framework

原因：
- 現代 .NET 專案更適合：
  - `dotnet build`
  - `dotnet test`
  - 現代 package / CLI / AI agent workflow
- 更符合自動化開發與測試需求
